SDGMart — How It All Works

A friendly, top-to-bottom tour of everything we built: the code, the choices behind it, the trade-offs, and every piece of technical machinery (domains, proxies, caches, payments) explained in plain language. Read it in order the first time; use the sidebar as a reference after.

Same-day grocery deliveryTamale, Ghana Web app + installable PWALive at sdg-mart.com

1 · The big picture

SDGMart is a web application — software that runs in a web browser instead of being installed like a phone app. A customer opens sdg-mart.com, browses groceries, places an order, and pays; a rider delivers it; an admin manages the whole thing from a dashboard.

Under the hood there are really only four kinds of machine doing the work, and everything else is plumbing that connects them:

PieceWhat it is (analogy)In SDGMart
The app (frontend)The storefront — what the customer sees and touchesReact code that runs in the browser
The server (backend)The shopkeeper in the back office — makes decisions, guards the rulesNode/Express (server.js)
The databaseThe filing cabinet — remembers everything permanentlySupabase (a Postgres database)
File storageThe photo album — holds images, which are too big for the filing cabinetSupabase Storage
Analogy
Think of a restaurant. The frontend is the dining room and menu. The backend is the kitchen and the manager who decides what's allowed. The database is the reservation book and recipe binder. File storage is the wall of framed photos. Customers only ever see the dining room — everything important and trustworthy happens in the back.

Every section below zooms into one stop on this trip.

3 · The app the customer sees (frontend)

The frontend is built with React — the most popular tool for building interactive web interfaces. Instead of one giant file, the screen is broken into reusable components: HomePage, CheckoutPage, CartDrawer, ProductPage, and so on. Each is a self-contained Lego brick that knows how to draw one part of the screen.

Why React
A grocery app is highly interactive — the cart updates live, search filters as you type, the map moves. Doing that with plain HTML would mean manually rewriting the page on every change. React lets you describe "what the screen should look like for this data," and it figures out the minimal update. It's the industry default, so help and libraries are abundant.
Alternatives we could have used
  • Vue / Svelte — similar idea, smaller communities. Svelte produces slightly faster code; React wins on ecosystem.
  • Plain HTML + a little JavaScript — fine for a brochure site, painful for a live cart and map.
  • A native phone app (iOS/Android) — best performance and app-store presence, but you'd build and maintain two separate apps plus the website. For a young business, one web app that installs to the home screen (a "PWA," §19) is far cheaper and reaches everyone instantly.

The unusual choice: "global-window" components, no build step in the browser

Most React projects use a heavy toolchain (imports, modules, a bundler config). SDGMart deliberately skips that. Each component file just defines its component and attaches it to the browser's global window object:

const CheckoutPage = ({ cart, setCart }) => { /* ...draw checkout... */ };
Object.assign(window, { CheckoutPage });  // now any file can use it

They reference each other as plain global names. React itself is loaded from a CDN (a public copy hosted on the internet) rather than installed.

Why this way
It keeps the project dead-simple to reason about — no module-resolution config, no framework magic. For a solo builder, "just a bunch of files that see each other" is easy to hold in your head. The trade-off (managing which name is defined where) is handled by keeping the file list small and ordered.

4 · The server / back office (backend)

The backend is a single Node.js program using the Express framework, all in server.js. Node lets you write server software in JavaScript (the same language as the browser code), so one language covers the whole project. Express is a thin helper for answering web requests.

The server's job is to answer API requests. An API ("Application Programming Interface") is just a set of URLs the frontend can call to get things done. Examples from SDGMart:

POST /api/orders            → create an order
GET  /api/me/orders         → list the signed-in customer's orders
POST /api/paystack/init     → start a payment
GET  /api/admin/retention   → the admin retention report
Analogy
The frontend is a customer at a counter; the API is the menu of things they're allowed to ask for. The customer can't walk into the kitchen (the database) directly — they can only order from the menu, and the shopkeeper (server) decides whether to honour each request.
Why one single server file / one process
SDGMart runs as one Node process. That's the simplest possible shape: easy to deploy, easy to debug, cheap to host. It comfortably handles a town-sized business. The main thing to remember is that anything the server keeps "in memory" (like our speed caches, §11) lives only in that one process — which is fine until you run multiple copies.
Alternatives
Microservices (many small servers, one per job) are what huge companies use — more scalable, far more complex. "Serverless" functions (tiny snippets that spin up per request) avoid managing a server but complicate things like our in-memory caches and background jobs. For SDGMart's scale, one modest server is the right call.

5 · The filing cabinet (database)

Permanent data lives in Supabase, which is a hosted PostgreSQL ("Postgres") database plus some extras. Postgres is a relational database: data lives in tables (like spreadsheets) with rows and columns, and tables can reference each other.

TableHolds
userscustomer & admin accounts
ordersevery order placed
productsthe catalogue
sessionswho's currently logged in
reviews, referrals, recurring_orders…one table per feature

All database access is funneled through one file, database.js. The rest of the code never talks to Postgres directly — it calls tidy helpers like db.orders.create(...) or db.users.get(id).

Why funnel everything through one file
If every corner of the app wrote its own database queries, a change to how orders are stored would mean hunting through the whole codebase. With one database.js, all data logic lives in one place — safer, and easy to find.
Why Supabase specifically
It gives you a real, reliable Postgres database plus file storage, authentication tools, and auto-generated APIs, on a generous free tier — without you having to run and back up a database server yourself. It's "batteries included," which is ideal for moving fast.
Alternatives
  • Firebase (Google) — similar all-in-one, but uses a "NoSQL" document model that's less natural for orders-and-relationships data.
  • Raw Postgres / MySQL on a server — full control, but now you're a database administrator (backups, updates, security) too.
  • MongoDB (NoSQL) — flexible, but relational data (a customer has orders which have items) is exactly what Postgres does best.

A key security detail: the "service key"

The server connects to Supabase using a powerful service key that bypasses row-level permissions. That's safe only because the key lives on the server, never in the browser. The customer's app can never touch the database directly — it must go through the server's API, which enforces the rules. This is the backbone of the whole security model. (§22)

6 · The photo album (file storage)

Databases are great for text and numbers but bad at big binary files like images. So product photos live in Supabase Storage — a place designed to hold files and serve them over the web. The database just stores a short text link to each photo (in the product's img column).

Why images are compressed to ~900px before upload
A photo straight from a phone can be 5 MB. Shrinking it to ~900 pixels / ~100 KB makes pages load fast on Tamale mobile networks and keeps you inside the storage & bandwidth ("egress") limits. We also added loading="lazy" so the browser only downloads a photo when you scroll to it — critical once the catalogue has hundreds of items.

7 · Where the server lives (Render)

Our server.js has to run on a computer that's always on and reachable from the internet. That computer is rented from Render, a hosting platform. Render's best trick: it watches our GitHub code repository, and every time we push new code, it automatically rebuilds and redeploys — no manual uploading.

What "build" and "deploy" mean here

  1. Build: Render downloads the latest code from GitHub, installs the dependencies (npm install), and gets everything ready.
  2. Deploy: Render starts server.js, which — at startup — bundles all the frontend files together (see §13) and begins answering requests.
Real example from this project
Two deploys once failed with git 403. That error happened during the download-from-GitHub step — Render's permission to read the repo had temporarily lapsed. It had nothing to do with our code (which built fine) or Cloudflare; a retry fixed it. Lesson: a "build failed" can be about fetching the code, not the code itself — always read which step failed.
Alternatives
Heroku (the original of this style, now paid), Railway/Fly.io (similar to Render), a raw virtual server from AWS/DigitalOcean (cheapest at scale, but you manage everything yourself), or Vercel/Netlify (superb for frontends, awkward for a long-running server with background jobs like ours).

8 · Domains & DNS (the internet's phone book)

Every computer on the internet has an IP address — a number like 216.24.57.1. Humans can't remember numbers, so we use domain names like sdg-mart.com. DNS (Domain Name System) is the global phone book that translates the name into the number.

You manage this translation with DNS records. The ones that matter:

RecordMeansExample use
Aname → an IP address (a number)point a domain straight at a server
CNAMEname → another namewww → sdgmart.onrender.com
TXTfree-text notes, used for verificationproving to Google/Resend you own the domain
The "@" you couldn't find
@ is DNS shorthand for "the bare domain itself" (sdg-mart.com with nothing in front). Cloudflare doesn't show a literal "@" — it just lists that row by the domain name. So the row named sdg-mart.com is your "@" record. The www row is the subdomain. Those two point at Render; the Google/Resend TXT records are unrelated verification notes and are left alone.

9 · Cloudflare & proxies (the shield in front)

A proxy is a middleman that stands between the visitor and your server. Instead of the customer's phone talking to Render directly, it talks to Cloudflare, and Cloudflare talks to Render. In the Cloudflare dashboard this is the difference between the grey cloud (DNS-only: Cloudflare just gives out the address and steps aside) and the orange cloud (proxied: all traffic flows through Cloudflare).

Grey cloudPhone → Render directly. Cloudflare only answered "where is it?"
vs
Orange cloudPhone → Cloudflare → Render. Cloudflare shields & speeds every request.

Turning on the orange cloud gives you three things for free:

  • CDN (speed): Cloudflare has servers worldwide. It keeps copies of your unchanging files (images, the app bundle) at a location near the visitor — so a shopper in Tamale gets them from nearby, not from Render's distant server. "CDN" = Content Delivery Network.
  • DDoS protection (safety): if someone floods your site with fake traffic to knock it over, Cloudflare's huge network absorbs it before it reaches your small server.
  • Hides your origin (safety): attackers see Cloudflare's address, not Render's, so they can't target your server directly.
The ordering trap we navigated
You must set the SSL mode to Full (strict) before flipping to orange. Flip to orange while SSL is on "Flexible" and you get an endless redirect loop — the site goes down. Sequence matters: SSL setting first, cloud colour second.
Why "proxy" also matters for privacy/DDoS in general
Because all traffic passes through one controlled checkpoint, you can add rules there (block bad countries, rate-limit abusers, cache aggressively) without touching your app. It's a single place to defend and optimise.

10 · HTTPS & SSL (the padlock)

The little padlock in the address bar means the connection is encrypted with SSL/TLS. Without it, someone on the same Wi-Fi could read everything sent — including passwords and payment details. With it, the data is scrambled so only the server can read it. Every serious site (and every payment provider) requires HTTPS.

"Full (strict)" — the Cloudflare setting — means: encrypt the visitor→Cloudflare hop and the Cloudflare→Render hop, and insist Render has a valid certificate. It's the most secure of Cloudflare's options and is safe because Render issues real certificates automatically.

Analogy
HTTP is a postcard — anyone handling it can read it. HTTPS is a sealed, tamper-proof envelope that only the recipient can open.

11 · Caching — the three layers (and why it bit us)

Caching means keeping a ready-made copy of something so you don't rebuild or refetch it every time. It's the single biggest source of "why isn't my change showing up?" confusion, because there are three separate caches in SDGMart's path, each owned by someone different:

LayerWho owns itWhat it stores
1. Server memoryour server.jsthe built app bundle, the catalogue, top-sellers — for speed under load
2. Cloudflare edgeCloudflarecopies of files near visitors
3. Browser cachethe visitor's phonefiles already downloaded once

Layer 1 — in the server

The catalogue file (/data/products.js) is loaded by every visitor on every page. Rebuilding it from the database each time was slow (~700 ms) and buckled under load. So the server now keeps the finished file in memory for 60 seconds, and admin edits instantly clear that cache. Result in load testing: from crashing at 20 simultaneous visitors to ~4,100 requests/second.

The trade-off to remember
This cache lives inside the one server process. If you ever run several copies of the server, each would have its own cache and they could briefly disagree. Fine on one machine (today's setup); something to redesign only if you scale out.

Layers 2 & 3 — the "stale for 4 hours" incident

When we turned on Cloudflare's orange cloud, a Cloudflare setting was silently overriding our server's instructions and telling browsers "keep this app code for 4 hours." So after a deploy, customers could keep running old code for up to 4 hours. We fixed it by setting Cloudflare to "Respect Existing Headers" — meaning Cloudflare obeys the cache duration our server sends (5 minutes for the app, "always check" for the catalogue and service worker) instead of overriding it.

Why "purge everything after each deploy" wasn't enough
Purging clears Cloudflare's copy (layer 2) but cannot reach the copies already sitting on customers' phones (layer 3). Those phones won't ask for a fresh copy until their own timer expires. The only real fix is at the instruction ("header") level, so phones are told a short duration in the first place. That's why the settings change — not a purge habit — was the correct solution.

12 · Why it's a "single-page app" (and the trade-off)

SDGMart is a Single-Page Application (SPA). The browser loads the app once, and from then on, moving between "pages" (home → product → checkout) just redraws the screen with JavaScript — no full reload. It feels instant, like a phone app.

 Single-page app (what we have)Multi-page (traditional)
FeelFast, app-likeEach click reloads the whole page
Search engines (SEO)Harder — one URLEach page indexable
Sharing a link to one productNeeds extra workNatural
Analytics per sectionNeeds manual eventsAutomatic
How we got the best of both
Rather than rebuild as multi-page, we added real URLs inside the SPA (/shop, /checkout, /order-confirmed…). The app still never fully reloads, but the address bar changes and each section has a shareable link — which also gave Google Analytics a proper page to track and Google Ads a real "order complete" address to count as a sale. Small change, most of the multi-page benefit, none of the rebuild.

13 · The bundling trick (esbuild)

Browsers can't directly run the modern, JSX-flavoured React code we write — and loading 20 separate files would be slow. So at server startup, a tool called esbuild stitches all the component files into one optimised file, /app.bundle.js, translated into plain browser-friendly JavaScript. The browser downloads just that one file (plus React from a CDN).

Why build once at startup, on the server
Doing the translation in the browser (an older approach called "Babel in the browser") made every visitor's phone do the work — slow. Building once on the server means the heavy lifting happens a single time, and every visitor gets the fast, finished result. The bundle is rebuilt only when a source file actually changes.
Alternatives
Big projects use Webpack or Vite with a separate build step during deployment. esbuild is dramatically faster and, done at startup, keeps the whole pipeline in one place with zero extra config — a good fit for this project's "keep it simple" philosophy.

14 · Accounts & staying logged in (sessions)

When you log in, the server checks your password (which is stored hashed — scrambled one-way so even we can't read it) and, if correct, creates a session: a long random token saved in the sessions table with a 7-day expiry. Your browser keeps that token and sends it with every request, in a header that says Authorization: Bearer <token>. The server looks it up to know who you are.

Analogy
The token is a coat-check ticket. Logging in hands you a numbered ticket; the matching stub stays with the server. Every future request, you flash the ticket and the server knows it's you — without you re-stating your name and password each time. Lose the ticket (log out) and it's torn up.
Why random tokens instead of sending the password each time
The password travels exactly once (at login). After that, only the meaningless random token moves around — so even if a single request were somehow exposed, your actual password isn't in it, and the token can be revoked (deleted) without changing your password.

15 · "Trust nothing from the browser" — server-side pricing

This is the most important security idea in the whole app. The browser code can be edited by anyone — a tech-savvy customer could change the price of an item to GHS 0.01 before checking out. So the server never trusts the prices the browser sends. When an order comes in, the server recomputes everything from scratch — real prices from the database, valid promos, the customer's actual loyalty balance, delivery fee — in a function called computeOrderPricing. That server-computed total is what gets charged and stored.

The golden rule
The frontend is for convenience and display; the backend is the source of truth. Anything that affects money or permissions must be decided on the server, because the browser can be tampered with. This single principle prevents a whole category of fraud.

The same principle appears everywhere: whether an order qualifies for free delivery, whether a birthday gift is valid, whether a guest can view a tracking page — all decided server-side, never taken on the browser's word.

16 · Clever stateless tokens (guest order tracking)

Guests (people who order without an account) still need to track their order later, even from a different phone. We could have stored a random secret code in the database for every order — but there's a neater way that stores nothing extra.

Each order gets a tracking token computed with HMAC — a mathematical "signature" that mixes the order's ID with a secret key only the server knows:

token = HMAC-SHA256(secret, "track:" + orderId)   → first 20 characters

To check a token, the server just recomputes it and compares. If they match, it's genuine. It's like a tamper-proof wax seal: only someone with the secret stamp could have produced it, and there's nothing to store or look up.

Why this is elegant
Zero database changes, no lookups, unguessable (20 hex characters ≈ 80 bits of randomness), and it survives server restarts because it's pure maths. The customer's phone keeps the token; a shared link carries it. We added a rule that it stops working 7 days after delivery, so an old shared link can't reveal someone's address forever.
Honest limitation
Because it's derived from the Supabase key, rotating that key would invalidate outstanding guest links. A future improvement is a dedicated TRACK_TOKEN_SECRET so the two are independent.

17 · Scheduled work without a scheduler (the /healthz trick)

Some jobs must run daily — send birthday wishes, place due recurring orders. Normally you'd use a cron job (a system timer). Render's free tier doesn't include one, so SDGMart uses a clever substitute.

An external monitoring service (UptimeRobot) already pings /healthz every few minutes to keep the free server awake. That endpoint runs the daily jobs — but guarded by a "last run date" marker so the actual work happens at most once per day, no matter how many pings arrive.

Analogy
It's like a shopkeeper who checks a wall calendar every time the doorbell rings: "Have I done today's opening tasks yet? No? Do them and tick the box. Already ticked? Ignore." The doorbell (the ping) is frequent; the tasks happen once a day.
Alternatives
A proper cron service (Render's paid tier, or a separate scheduler) is cleaner and fires at an exact time. The /healthz trick is a pragmatic zero-cost stand-in that's served the project well; worth upgrading if timing ever needs to be precise.

18 · Payments & webhooks (Paystack)

SDGMart never handles card or mobile-money details itself — that would require heavy security certification. Instead it uses Paystack, a Ghanaian-friendly payment provider. At checkout, Paystack opens its own secure popup; the customer pays there; SDGMart only learns the result.

The flow uses a webhook — a way for Paystack to call us when something happens (rather than us constantly asking "is it paid yet?"):

1. initServer asks Paystack to start a payment
→
2. popupCustomer pays in Paystack's secure window
→
3. webhookPaystack calls our server: "paid!"
→
4. orderServer creates the paid order
Why a webhook AND a "verify" step
Two independent confirmations make it robust: if the customer's browser confirms, great; if their tab closes mid-payment, the webhook still tells us and the order is created anyway. The server checks Paystack's cryptographic signature on the webhook so nobody can fake a "paid" message.
The OTP note we added
The one-time password (OTP) for mobile money is sent by the customer's bank/telco, not by SDGMart — so if their network is slow, the SMS lags. We added an on-screen note telling customers to wait about a minute for Paystack's "get OTP via WhatsApp" backup. It's a real-world UX fix for a Ghana-specific network reality.

19 · Installable app & notifications (PWA + Web Push)

SDGMart is a PWA (Progressive Web App) — a website that can be "installed" to a phone's home screen and then behaves like a native app (full-screen, its own icon, works partly offline). The magic ingredients are a manifest.json (name, icon, colours) and a service worker (sw.js) — a small script the browser runs in the background even when the site isn't open.

The service worker also enables Web Push notifications ("Your rider is on the way!"). These use VAPID keys, a standard that lets a server send a notification to a specific browser securely.

The iPhone caveat worth knowing
On iPhone, push notifications only work if the customer installs the app to their home screen first (Apple's rule). In plain Safari they get no pushes. That's why nudging installation matters most for iPhone users — it's the gateway to notifications. Android has no such restriction.

20 · Feature deep-dive: Squads & loyalty

"Squads" are a group-buying incentive: invite friends with your referral link, and when every squad member's spending crosses GHS 500, everyone gets a GHS 25 loyalty reward. It's a growth engine — customers recruit customers.

The rollover fix we made

Originally, when a squad hit the goal, everyone's counter reset to zero — so a member who'd reached GHS 730 while waiting for slower friends lost the extra 230. Now that overflow rolls into the next round: they restart at 230, not 0. It rewards the eager shopper instead of punishing them, and it's simply fairer.

The 5-member cap

Squads are limited to 5 members. A 6th person using your link still counts as your referral (you get credited) but starts their own squad. This keeps the "everyone must hit 500" goal achievable rather than impossible with a huge group.

Design lesson
Incentive systems have to be tuned so the goal feels reachable and effort feels rewarded. Both fixes came from thinking through the human experience of a real squad, not just the code.

21 · Feature deep-dive: recurring orders

Customers can set up an auto-reorder ("my usual, every 2 weeks"). The interface and data existed for a while, but nothing actually placed the orders — so we built the job that does. Each day (via the /healthz trick, §17) it finds recurring orders that are due and places them through the exact same code path as a normal checkout — so pricing, promos, loyalty, and the delivery cutoff all behave identically.

It handles the messy real-world cases carefully:

  • Items that are sold out or no longer exist are dropped, with a "we skipped this round" notification if that empties the order.
  • It's always Cash on Delivery — there's no stored card to auto-charge (Paystack is popup-only, needs the customer present).
  • The next date advances even if a run fails, so a broken order can't retry forever and spam the customer daily.
  • Each order is wrapped in error handling, so one failure doesn't stop the others.
Design lesson
Automated jobs run unattended, so they must fail gracefully and quietly move on. Half of this feature is the happy path; the other half is "what if the product's gone / the run errors / it's not due yet" — and that half is what makes it trustworthy.

22 · The security model in one picture

Every protection in SDGMart follows from one principle: the browser is untrusted; the server is the guardian.

ProtectionHow it works
Prices can't be fakedServer recomputes every total from the database (§15)
Database can't be touched directlyPowerful key lives only on the server; browser must go through the API (§5)
Passwords can't leakStored hashed (one-way scrambled); only a random token travels after login (§14)
Guest tracking can't be guessedUnguessable signed token, expires after delivery (§16)
Fake "payment done" messagesWebhook signature checked cryptographically (§18)
Source code & secrets stay hiddenServer blocks direct download of code/config files; secrets live in environment variables, never in the repo
Traffic is encrypted & shieldedHTTPS everywhere + Cloudflare in front (§10, §9)

24 · How SDGMart could grow (and what to watch)

  • Traffic grows: upgrade Render (bigger machine, so requests don't queue) and Supabase (more bandwidth for images + automatic backups). Neither needs code changes.
  • If you ever run multiple server copies: the in-memory caches (§11) would need to move to a shared store like Redis, since each copy currently caches independently.
  • Precise scheduling: replace the /healthz daily-job trick with a real cron service if timing ever needs to be exact.
  • Search & discovery: as the catalogue grows to hundreds of items, a few dedicated, indexable product pages would help Google send you shoppers.
  • Resilience: a dedicated tracking-token secret, automated tests, and an error-monitoring service (a Sentry hook already exists) are natural next hardening steps.
